# Local Coding Model Bake‑off

This repository hosts a small but carefully designed benchmark for comparing four large language models as on‑device coding assistants.  The goal was to answer a very practical question: **when my quota for frontier LLMs like Codex runs out, which locally run model should I trust to help me debug, refactor and write code?**

We evaluated and compared the following four open models:

* **Qwen3.5‑27B (Dense)** — a dense transformer from Alibaba’s Qwen series.
* **Gemma 4 31B (Dense)** — Google’s 31 billion‑parameter dense Gemma model.
* **Qwen3.5‑30B‑A3B (MoE)** — a mixture‑of‑experts version of Qwen built for speed and efficiency.
* **Gemma 4 27B (MoE)** — the 27 billion‑parameter MoE variant of Gemma 4.

Throughout this document we refer to the two dense models as **Qwen Dense** and **Gemma Dense**, and the two mixture‑of‑experts models as **Qwen MoE** and **Gemma MoE**.

## Bottom line

After running all four models through three complementary debugging challenges, we concluded that:

* **Dense models are consistently more trustworthy than MoE models**.  MoE variants were faster but more likely to overgeneralize, misattribute the bug to the wrong subsystem, or propose overly broad patches.
* **Qwen Dense and Gemma Dense are both very capable**, but their strengths differ.  Qwen Dense excelled on the deepest multi‑file debugging task and currently has the stronger published coding benchmarks, while Gemma Dense showed a conservative bias toward minimal, production‑safe fixes and excelled on the UI‑centric challenge.
* For a single daily driver, we recommend **Qwen Dense** because of its balanced performance across tasks and public evidence on coding benchmarks.  **Gemma Dense** is an excellent second opinion for cross‑checking patches and offers a complementary reasoning style.
* **Use MoE models as fast, low‑cost assistants**—they are suitable for brainstorming and generating draft ideas but should not be the final authority for critical fixes.

## Benchmark design

The benchmark consists of three handcrafted challenges, each probing a different axis of practical debugging and coding assistance.  We used separate weighted scores for each task and summed them into a final ranking.

### Task 14958 — Validation / spec reasoning

* **Purpose:** Test the model’s ability to identify and explain a subtle validation regression, such as an input that is accepted when it should not be.
* **What it measures:** Structured diagnostic reasoning, familiarity with input validation patterns, and the discipline to propose a small, safe fix rather than rewriting everything.
* **Role in suite:** Independent corroboration task that helps confirm consistency across models.  It counts less than the other two because it is easier to solve with generic pattern‑matching.

### Task 15241 — Non‑vision visual‑debugging inference

* **Purpose:** Simulate a visual bug in a mobile UI without requiring screenshots or vision.  The agent must infer why a button becomes unclickable and propose the minimal CSS fix.
* **What it measures:** Ambiguity handling, reasoning about hit‑testing and element overlap, and ability to scope fixes to the relevant viewport (for example, mobile only).
* **Role in suite:** Tests minimal‑fix discipline and rewards agents that stay close to the problem description.  Weighted moderately in the final score.

### Task 15273 — Optimistic UI / offline retry / reconciliation

* **Purpose:** Diagnose a bug where optimistic comments are duplicated after offline retries succeed.  The patch includes an inadvertent removal of a `clientID` parameter, and the agent must trace how missing identity propagates through reconciliation.
* **What it measures:** Deep causal reasoning across multiple files, ability to follow asynchronous flows, and differentiating between a containment fix and a true root fix.
* **Role in suite:** The strongest discriminator.  Weighted heaviest because it forces the model to connect the bug to the specific changed line and propose the smallest fix that actually addresses the root cause.

### Why Task 15193 was discarded

We originally included a fourth challenge but removed it after discovering that its specification was generated by a local model and that all four submissions solved a wrong subsystem.  Rather than over‑fit to a flawed task, we replaced it with Task 15273 and recomputed all scores.

## Evaluation methodology

Each submission was graded by a human reviewer (Pat Narendra, PhD) on four criteria:

1. **Root‑cause diagnosis:** How accurately did the model identify the true cause of the bug?
2. **Fix correctness and minimality:** Did the model propose the smallest patch that resolves the issue without side‑effects?
3. **Verification quality:** How well did the model specify tests to confirm the fix and prevent regressions?
4. **Harness discipline:** Did the model follow the task instructions and deliver the expected files and formats?

We weighted the tasks as follows: **15273** (45%), **15241** (35%), **14958** (20%).  Dense and MoE variants were run in identical harness environments to ensure fairness.

## High‑level results

Across the three tasks, the pattern was consistent:

* **Dense models outperformed MoE models.**  They were more reliable on pinpointing the actual bug, tended to write smaller patches, and followed the harness instructions more faithfully.  MoE models were faster but more prone to plausible‑but‑wrong stories.
* **Qwen Dense vs Gemma Dense:** No model dominated every task.  Gemma Dense won Task 15241 with its conservative CSS fix, while Qwen Dense won Tasks 14958 and 15273 by anchoring its diagnosis to the right file and providing the cleanest root fix.  Taking the weighted average, **Qwen Dense edges out Gemma Dense** as the best overall daily driver.
* **Qwen MoE vs Gemma MoE:** Qwen’s MoE variant performed better than Gemma’s MoE, correctly identifying the broken request in Task 15273 and ranking second overall on that challenge.  However, both MoE models trailed their dense counterparts significantly.

## Final rankings

1. **Qwen3.5‑27B (Dense)** — Best overall and recommended daily driver.  Strong on deep debugging tasks, with balanced reasoning and disciplined patch proposals.
2. **Gemma 4 31B (Dense)** — Reliable and conservative.  Best for cross‑checking fixes and catching over‑broad patches.  Complementary reasoning style to Qwen.
3. **Qwen3.5‑30B‑A3B (MoE)** — Fast, useful for brainstorming and draft code, but less trustworthy for final patches.  Impressively strong on the deepest task for an MoE model.
4. **Gemma 4 27B (MoE)** — Adequate but least consistent; tends to provide over‑general explanations and to misattribute bugs.  Use only with supervision.

## Practical recommendations

For real development work:

* **Use Qwen Dense as your primary local coding assistant.**  It offers the best balance of reliability, depth and performance.
* **Keep Gemma Dense around as a second opinion.**  When Qwen’s answer feels speculative or over‑confident, Gemma’s conservative style can catch issues.
* **Leverage MoE models for speed.**  Both Qwen MoE and Gemma MoE are useful for scaffolding, brainstorming or generating test cases.  Always verify their outputs with a dense model before shipping a patch.
